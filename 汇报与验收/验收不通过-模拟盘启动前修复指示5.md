## 必修 2：启动模拟按钮必须真正启动服务

当前 `btc5m-sim.service` 是 inactive。

WebUI 的 `/api/toggle` 目前只是改配置：

```python
cfg["paused"] = (act == "pause")
```

它不会执行：

```bash
systemctl --user start btc5m-sim.service
systemctl --user stop btc5m-sim.service
```

所以用户点击“启动模拟”后，配置可能变了，但模拟进程不会运行。

要求：

1. 点击“启动模拟”时，如果 `btc5m-sim.service` 不在运行，后端必须启动它。
2. 点击“暂停模拟”时，可以先只写 `paused=true`，也可以停止服务，但 WebUI 状态必须真实反映。
3. `/api/safety` 和 `/api/status` 里的 `sim_active / service_active / sim_trading` 必须一致。
4. 启动失败要在 WebUI 明确提示，不能假装已启动。

建议在 `api_server.py` 的 `toggle()` 里对 sim 模式调用：

```python
systemctl --user start btc5m-sim.service
```

注意只允许启动模拟服务，不要启动实盘自动交易。

## 必修 3：WebUI 交易记录 API 必须兼容新字段

文件：

`C:\Users\yyq\Desktop\自动交易\btc5m-webui\api_server.py`

当前 `/api/trades` 仍按旧字段读：

```python
"btc_open": float(t.get("ptb", 0))
"buy_prob": float(t.get("buy_prob", 0))
"buy_amount": float(t.get("buy_amount", 0))
"net_profit": float(t.get("net_profit", 0))
"settle_gap": float(t.get("gap", 0))
```

但新模拟记录写的是：

```json
{
  "open_price": 76816.36,
  "platform_open_price": 76816.36,
  "probability": 0.455,
  "amount": 1.0,
  "pnl": 0.8,
  "return": 0.8,
  "settlement_gap": -181.08,
  "settlement_status": "confirmed"
}
```

必须做兼容映射：

```python
btc_open = t.get("open_price", t.get("platform_open_price", t.get("ptb", 0)))
buy_prob = t.get("probability", t.get("buy_prob", 0))
buy_amount = t.get("amount", t.get("buy_amount", 0))
net_profit = t.get("pnl", t.get("net_profit", 0))
return_pct = t.get("return_pct")
if return_pct is None and t.get("return") is not None:
    return_pct = float(t["return"]) * 100
settle_gap = t.get("settlement_gap", t.get("settle_gap", t.get("gap", 0)))
```

`btc_final` 不能这样写：

```python

```

因为 pending 记录的 `btc_final` 是 `None`，WebUI 会显示成 `$0` 或直接报错。要求：

1. pending 时返回 `btc_final=null`。
2. 前端显示“待平台结算”，不要显示 `$0`。
3. `settlement_status` 必须返回给前端。
4. `settle_source` 必须返回给前端。
5. `exclude_from_backtest` 必须返回给前端。

## 必修 5：汇总统计只能统计有效记录

当前 `/api/summary` 仍按旧字段：

```python
won = sum(1 for t in trades if t.get("won"))

```

新记录不会稳定提供 `won` 或 `net_profit`。

1. 只统计 `settlement_status == "confirmed"`。
2. 排除 `exclude_from_backtest == true`。
3. `won` 用 `status == "won"`。
4. `lost` 用 `status == "lost"`。
5. 盈亏优先读 `pnl`，兼容旧字段 `net_profit`。
6. 投入优先读 `amount`，兼容旧字段 `buy_amount`。
7. pending 记录只进“待结算数”，不进胜率和盈亏。

每日汇总页也必须同样处理。

## 必修 6：不要让 pending 记录永久丢失

现在 `sim/trader.py` 如果平台 `closePrice` 暂时没有返回，会写一条 pending，然后 `processed.add(slug)`，之后不会再补结算。

这不符合 24 小时模拟盘要求。

必须实现至少一种方案：

方案 A：结算时等待平台 closePrice，最多等 10 分钟，拿到后才写 confirmed 记录。

方案 B：先写 pending，但后台 pending resolver 定期读取 pending 记录，拿到 `closePrice` 后更新为 confirmed。

推荐先做方案 A，简单可靠：

```python
platform_price = fetch_platform_crypto_price(window_start_ts, window_end_ts, max_retries=300, retry_interval=2.0)
```

如果 10 分钟仍没有 closePrice，再写 `fallback_missing`，并 `exclude_from_backtest=true`。

不要把 pending 当成一笔已完成交易。
