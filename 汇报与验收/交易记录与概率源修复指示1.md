# 交易记录与概率源修复指示1

## 背景

Codex 已在 13:20 后完成以下改动：

1. 模拟盘入场逻辑改成“进入 T-N 秒窗口后持续监控”，不是固定 T-25 打点。
2. 策略入场窗口范围改成 1-120 秒。
3. 采集器 Gamma `/markets` 失败时增加 `/events` 兜底。
4. WebUI 交易记录表已按截图重做，并加入买入价差、结算价差。
5. WebUI 后端 `/api/trades` 已暴露 `buy_gap`、`settlement_gap`、`fee`、`return_pct` 等字段。

## 当前最重要问题

交易记录里仍出现“概率/买入价不可信”的情况，例如：

- `btc-updown-5m-1779862200`
- 方向显示 Up
- 但买入价/概率出现 `$0.010` 或 `1%`
- 同时平台 Gamma 显示 outcomePrices 为 `["0.025", "0.975"]`
- 这说明可能存在 token 映射、盘口方向、概率来源或跳过记录写入逻辑错误。

必须先解决这个问题，否则后续回测和模拟盘都不可信。

## 第一任务：确认 Up/Down token 映射

检查以下文件：

- `btc5m-trader/collectors/true_market_collector.py`
- `btc5m-trader/sim/trader.py`
- `btc5m-webui/api_server.py`

要求：

1. 从 Gamma 获取市场时必须读取：
   - `outcomes`
   - `clobTokenIds`
   - `outcomePrices`
2. 不允许默认假设 `clobTokenIds[0]` 一定是 Up，必须根据 `outcomes` 映射：
   - outcomes 中 `"Up"` 对应的 token 才是 `token_up`
   - outcomes 中 `"Down"` 对应的 token 才是 `token_down`
3. `windows.jsonl` 里要保存：
   - `outcomes`
   - `token_up`
   - `token_down`
   - `token_map_source`
4. 若无法解析 outcomes，必须标记：
   - `token_map_quality="bad"`
   - `exclude_from_backtest=true`
   - 不允许用于模拟交易入场。

## 第二任务：修复盘口概率来源

检查 `latest_entry_quote()`。

要求：

1. 买 Up 时必须取 Up token 的 ask/market buy 模拟价格。
2. 买 Down 时必须取 Down token 的 ask/market buy 模拟价格。
3. `probability` 字段必须是入场时真实可买价格，不是开盘 outcomePrices。
4. 如果 `probability_source="market_open_fallback"`，该记录必须：
   - `exclude_from_backtest=true`
   - 不能作为有效回测样本
   - WebUI 说明里显示“盘口缺失，使用开盘概率兜底”
5. 不允许用单边 `price_change` 直接推导完整概率，除非已经维护了完整内存订单簿。

## 第三任务：修复跳过记录

现在出现 `skip_reason=waiting_for_signal` 但同时有很高概率和大价差的记录，这不合理。

要求：

1. 每次窗口结束跳过时，`skip_reason` 必须是最后一次真实失败条件：
   - 价差不足
   - 概率不足
   - 盘口缺失
   - 价格不可执行
   - PTB 缺失
2. 不允许最终写 `waiting_for_signal`。
3. 若窗口内曾经满足价差但概率不满足，要写概率不足。
4. 若窗口内曾经盘口缺失，要写盘口缺失。
5. 跳过记录也必须保留：
   - `buy_gap`
   - `settlement_gap`
   - `probability`
   - `probability_source`
   - `entry_seconds_before`
   - `monitor_evaluations`
   - `best_probability_seen`
   - `best_gap_seen`

## 第四任务：修复交易记录字段完整性

检查 `/api/trades` 返回字段。

要求每条记录必须包含：

- `status`
- `time`
- `window_start_ts`
- `market_time`
- `direction`：买入方向，不是结算方向
- `settlement_direction`：结算方向
- `btc_open`
- `btc_entry`
- `btc_final`
- `buy_gap`
- `settlement_gap`
- `buy_prob`
- `buy_amount`
- `fee`
- `entry_seconds_before`
- `net_profit`
- `return_pct`
- `skip_reason`
- `probability_source`
- `exclude_from_backtest`

## 第五任务：WebUI 展示要求

交易记录表列保持 Codex 已改的顺序：

状态 / 时间 / 市场 / 方向 / 开盘价 / 买入BTC / 结算价 / 买入价差 / 结算价差 / 买入价 / 买入金额 / 手续费 / 入场 / 盈亏 / 盈亏率 / 说明

展示要求：

1. 买入价差、结算价差：
   - 正数绿色
   - 负数红色
2. 买入BTC：
   - 橙色加粗
3. 市场 ID：
   - 蓝色
4. 盈亏/盈亏率：
   - 正数绿色
   - 负数红色
   - 0 或无交易显示 `--`
5. 跳过记录说明必须可读，不能只显示内部英文 `waiting_for_signal`。

## 第六任务：验收方式

完成后必须创建新文件：

`汇报7.md`

汇报中必须包含：

1. 改动文件列表。
2. 用最近 5 条 `/api/trades` 返回 JSON 证明字段完整。
3. 抽查至少 3 个市场：
   - Gamma outcomes
   - clobTokenIds
   - token_up/token_down 映射结果
   - 对应盘口 up/down ask
   - 最终写入交易记录的 buy_prob
4. 证明不会再出现：
   - Up 方向但使用 Down 价格
   - Down 方向但使用 Up 价格
   - `waiting_for_signal` 作为最终跳过原因
   - `return_pct` 被覆盖成 0
5. 说明当前数据是否可以继续跑 24 小时模拟盘。

## Codex 判断

现在先不要开启正式 24 小时有效数据统计。交易记录 UI 已经明显好很多，但“概率源/方向映射”还没完全验收，必须先修掉。这个问题不解决，后面跑一晚上数据也可能是假数据。
