# 回测真实数据采集方案

目标：让模拟盘和回测尽量复刻 Polymarket 真实交易环境。原则是先用 Polymarket 官方数据；官方没有历史秒级数据的地方，从现在开始自己采集，不能用 Binance/普通 Chainlink Feed 事后补。

## 先纠正几个结论

1. `实盘余额` 指后端实盘钱包里当前可用于下单的 pUSD。用户转入约 2 美元后，已经用约 1 美元做了一笔真实测试单，所以当前可下单余额约为 1.057971 pUSD。
2. 买入后的钱会变成 Up/Down 份额，不再显示为现金余额。若那笔单最终赢，结算/赎回后才回到 pUSD；若输，份额归零。
3. 回测不能只看 BTC 价格。真正影响成交的是 Polymarket 的盘口：best ask、盘口深度、成交价、滑点、当时是否有足够可买量。
4. Gamma API 能补市场元数据和当前快照，但不能提供历史逐秒盘口。逐秒盘口必须从 CLOB WebSocket 实时保存。

## Hermes 现在要做的采集器

建立一个独立采集服务，例如 `btc5m-trader/collectors/true_market_collector.py`，只采集，不交易。

每个 5 分钟市场创建一个 `market_window_id`，格式建议：

```text
{asset}_{timeframe}_{window_start_ts}
btc_5m_1779788700
```

每秒或每次事件写入这些 jsonl：

```text
btc5m数据/true_market/windows.jsonl
btc5m数据/true_market/price_ticks.jsonl
btc5m数据/true_market/orderbook_ticks.jsonl
btc5m数据/true_market/trade_ticks.jsonl
btc5m数据/true_market/market_meta.jsonl
btc5m数据/true_market/resolutions.jsonl
```

## 数据源分级

### A 级：平台真实数据，必须优先

1. Gamma API
   - 用途：发现市场、拿 slug、condition_id、token_ids、outcomePrices、bestBid、bestAsk、lastTradePrice、orderMinSize、tickSize、feeSchedule、resolutionSource、volume、liquidity、start/end time。
   - 写入：`market_meta.jsonl`，每个市场至少保存 open、T-60、T-30、T-10、close 后各一次。

2. CLOB WebSocket market channel
   - endpoint：`wss://ws-subscriptions-clob.polymarket.com/ws/market`
   - 订阅：当前市场 Up/Down 两个 token_id。
   - 必须设置：`custom_feature_enabled: true`
   - 采集事件：`book`、`price_change`、`best_bid_ask`、`last_trade_price`、`market_resolved`。
   - 写入：`orderbook_ticks.jsonl` 和 `trade_ticks.jsonl`。
   - 这是回测“能不能成交、成交价是多少”的核心。

3. CLOB REST
   - `/book` 或 `/books`：在新市场开始、入场前、入场时、结算前抓完整盘口快照。
   - `/prices-history`：只能做分钟级历史价格补充，不能代替秒级盘口。
   - `/clob-markets/{condition_id}`：补最小下单、tick size、手续费参数。

4. Polymarket RTDS Chainlink 流
   - endpoint：`wss://ws-live-data.polymarket.com`
   - topic：`crypto_prices_chainlink`
   - filters：`{"symbol":"btc/usd"}`，后续扩展 `eth/usd`、`sol/usd`、`xrp/usd`。
   - 写入：`price_ticks.jsonl`
   - 每条必须保存 `payload.timestamp`、本机 `received_at`、`value`。

5. 平台结算结果
   - 优先从 Gamma closed market / resolved market 字段、CLOB `market_resolved` 事件、持仓/赎回结果三方交叉确认。
   - 写入：`resolutions.jsonl`
   - 字段必须包括：`actual_winner`、`resolved_at`、`source`、`condition_id`、`winning_token_id`、`payout`。

### B 级：平台没有历史，只能自己从现在开始采

1. 秒级概率时间序列
   - 来源：CLOB WS 的 best ask/bid、price_change、best_bid_ask。
   - 保存 Up/Down 的 bid1、ask1、mid、spread、前 5 档深度。

2. 秒级盘口深度
   - 来源：CLOB WS 维护本地 orderbook，定时落盘完整快照。
   - 最少每秒保存一次；入场前后 10 秒建议保存每个事件。

3. 精确入场滑点
   - 回测时不能直接用 `outcomePrices`，要用当时 ask 队列模拟 $1 market buy，算平均成交价和剩余未成交量。

### C 级：只能作为参考，不准当真实回测依据

1. Binance BTC/USDT
   - 可用于辅助观察行情，不可用于判定 Polymarket 赢家。

2. 普通 Chainlink `latestRoundData`
   - 可用于辅助价差监控，但不等于平台最终结算流。

3. 第三方数据商
   - 可用于补历史研究，但接入前必须标记来源，不得混进“平台真实数据”。

## `btc_final` 应该怎么拿

不要用 Binance，也不要只用普通 Chainlink Feed。

执行顺序：

1. 实时采集 RTDS `crypto_prices_chainlink`，保存窗口结束前后至少 10 秒的 tick。
2. 窗口结束后，从平台 resolved 结果确认赢家。
3. 回填 `btc_final` 时，选与窗口结束时间最接近、且时间戳不晚于平台判定点的 Chainlink RTDS tick。
4. 若 `btc_final` 计算赢家和平台 `actual_winner` 不一致，以平台 `actual_winner` 为准，并把该窗口标记为 `settlement_mismatch=true`。

## `actual_winner` 应该怎么拿

执行顺序：

1. 监听 CLOB WebSocket `market_resolved`。
2. 轮询 Gamma API 的 closed/resolved market 字段，保存完整 market JSON。
3. 查询自己账户真实订单/持仓结算结果：赢的 token 最终可以赎回 1 pUSD，输的为 0。
4. 必要时再查 UMA/CTF 链上事件，但这一步先作为校验，不作为第一版依赖。

## Hermes 第一阶段任务

1. 停止改策略，先写采集器。
2. 当前只采 BTC 5M，但代码里的 asset/timeframe 要支持 BTC/ETH/SOL/XRP 和 5m/15m。
3. 每条数据都加：
   - `source`
   - `server_ts`
   - `received_at`
   - `local_monotonic_ms`
   - `market_window_id`
   - `slug`
   - `token_id`
4. WebUI 增加一个“数据质量”区：
   - 当前市场是否拿到完整 token_id
   - WebSocket 是否在线
   - 最近盘口 tick 延迟
   - 最近 Chainlink RTDS tick 延迟
   - 当前窗口数据完整度
5. 采满至少 24 小时 BTC 5M 后，再跑新回测。
